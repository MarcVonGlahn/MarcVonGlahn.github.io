Overview

This casual motorsport management game was the centerpiece of my Bachelor's Thesis in Game Engineering, and was developed solo in Unity 3D. The biggest emphasis was placed on the AI programming, making strong use of concepts like Finite State Machines and Utility Based AI. However, I also handled the gameplay programming, 3D modelling, shader programming using Unity Shader Graph, as well as the implementation of sound effects using Wwise.

I wanted players to experience motorsport like never before: very simple. In this exciting, fast-paced management game you can be team principal of your own team, upgrade your car with newly built parts and train your drivers to compete in races and win the championship. Use your drivers' Special Abilities to gain an advantage over other drivers on track and leave them in the dust.

Game AI

The AI system in Flat Spot is the core of the racing experience. Each driver on the track is controlled by an AI that makes decisions about racing lines, overtaking, defending, and when to use their Special Abilities. The AI is built on two main pillars: Finite State Machines for structured behavior flow, and Utility Based AI for dynamic decision-making within those states.

Finite State Machines

Each driver AI operates through two Finite State Machines that defines the high-level behavior during a race. The Speed_FSM is responsible for controling states like Acceleration, Braking, Holding and Pitlane. The Line_FSM on the other hand is responsible for the racing line the car takes, which in Flat Spot is defined through the 3 tracks Ideal, Outside and Inside. This structure ensures predictable and maintainable AI behavior while still allowing for dynamic responses to race conditions.

The State Machine has a reference to the current state, for which it executes the 3 main functions OnStateEnter(), OnStateExit() and OnStateRunning():

public class StateMachine : MonoBehaviour
{
    [Header("ShowLog")]
    [SerializeField] private bool showLog;

    protected State currentState;

    private void Update()
    {
        if (currentState == null)
            return;

        currentState.OnStateRunning();
    }

    public void SetState(State newState)
    {
        if (currentState != null)
            currentState.OnStateExit();

        currentState = newState;

        if (currentState != null)
            currentState.OnStateEnter();
    }

    protected void LogMessage(string message)
    {
        if (!showLog)
            return;

        Debug.Log($"{gameObject.name}: {message}");
    }
}

All of the states in an implemented state machine are wrapped inside a partial class for maximum scope control of the states. An example state machine setup for a Speed FSM would look like this:

public partial class Speed_RacingStateMachine : MarcimanStateMachine
{
    private float _timeInCurrentState;

    private float _currentSpeed = 0f;

    public float CurrentSpeed { get => _currentSpeed; }


    #region Events
    public delegate void EnterBreakingState();
    public event EnterBreakingState OnEnterBreaking;

    public delegate void ExitBreakingState();
    public event ExitBreakingState OnExitBreaking;
    #endregion


    private void Start()
    {
        ChangeState(RacingSpeedState.Idle);
    }

    private void ChangeState(RacingSpeedState newState)
    {
        _currentState = newState;

        switch (newState)
        {
            case RacingSpeedState.Idle:
                SetState(new Speed_RacingState_Idle(this));
                break;
            case RacingSpeedState.Accelerate:
                SetState(new Speed_RacingState_Accelerating(this));
                break;
            case RacingSpeedState.Breaking:
                SetState(new Speed_RacingState_Breaking(this));
                break;
            case RacingSpeedState.Hold:
                SetState(new Speed_RacingState_Holding(this));
                break;
            case RacingSpeedState.PitLane:
                SetState(new Speed_RacingState_PitLane(this));
                break;
        }
    }
}

public partial class Speed_RacingStateMachine
{
    public class Speed_RacingState_Accelerating : Speed_RacingState_Base
    {
        public Speed_RacingState_Accelerating(Speed_RacingStateMachine manager) : base(manager)
        {

        }

        public override void OnStateEnter()
        {
            base.OnStateEnter();

            EvaluateCurrentSpeed_OnEntry();
        }

        public override void OnStateRunning()
        {
            base.OnStateRunning();

            EvaluateCurrentSpeed_OnRunning();

            if (CheckIfBreakingReached())
            {
                Manager.ChangeState(RacingSpeedState.Breaking);
            }
        }

        public override void OnStateExit()
        {
            base.OnStateExit();
        }

        #region Override Evaluate Speed
        protected override void EvaluateCurrentSpeed_OnEntry()
        {
            float temp = GetTimeFromAnimationCurveValue();

            speedOnStateEntry = Manager._currentSpeed;

            timeFractionPassedOnEntry = temp;
        }

        protected override void EvaluateCurrentSpeed_OnRunning()
        {
			// Evaluating Acceleration and current speed based on the Properties of both driver and car.
            float lerp = Manager._timeInCurrentState / (Manager.inRacePropertiesManager.RaceCarProperties.TimeToFullSpeed - Manager.GetTyreDegBonus());
            float eval = Manager.inRacePropertiesManager.RaceCarProperties.AccelerationCurve.Evaluate(lerp + timeFractionPassedOnEntry);

            Manager._currentSpeed = eval * (Manager.inRacePropertiesManager.TopSpeed + GetTopSpeedBonus());
        }
        #endregion


        private float GetTopSpeedBonus()
        {
            float topSpeedBonus = 0f;

			// Special Logic based on Upgrades in the game.

            return topSpeedBonus;
        }
    }
}

As mentioned, there are multiple State Machines in the game, all of which tie together to bring the cars in Flat Spot to life. I am proud to say that for my first commercial project, this game scaled quite well, especially thanks to the robust base architecture of the State Machine.

Utility Based AI

In Flat Spot, the player takes the role of the manager. To have AI opponents in the game, I constructed a basic Utility-Based AI logic to create AI-managers of the other teams as your opponents.

UI Programming

As a management game, Flat Spot relies heavily on its user interface to communicate race information, team management options, and progression systems to the player. The UI was built entirely in Unity using the built-in UI system, with custom scripts handling data binding, state management, and dynamic updates during races.

Key UI systems include the race dashboard showing real-time positions and lap times, the team management screen for upgrading cars and training drivers, and the championship standings overview. Each panel is driven by dedicated controller scripts that pull data from the core game systems and update the display accordingly.

Audio Implementation with Wwise

The audio experience in Flat Spot was implemented using Audiokinetic's Wwise middleware. Wwise allows for dynamic and context-aware audio that responds to gameplay events, such as engine sounds changing with RPM, crowd reactions based on race position, and impactful sound effects when Special Abilities are activated.

The integration with Unity involves setting up Wwise events and RTPCs (Real-Time Parameter Controls) that are triggered from C# scripts. This approach separates audio design from code, allowing for iteration on sound design without needing to modify the game code.
