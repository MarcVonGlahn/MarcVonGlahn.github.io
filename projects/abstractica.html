ABSTRACTICA

Third Person Shooter with Roguelike Elements

This project is still on-going, thus this page will be updated regularly. Some sections are still under construction.

Key Features

  • Third person shooter with roguelike elements and permanent progression.
  • Custom Manager Registry System for enemy spawning, global material management, combat and other system.
  • Highly stylized world with no lit material and an expanding color change shader using Signed Distance Fields.
  • Gameplay Ability System for skill and ability management

Overview

This is a third person shooter with roguelike elements, devoloped by myself in Unreal Engine 5. The project started as a learning experience for myself to deepen my knowledge in UE5, and then naturally progressed into a full game that is available to wishlist on Steam.

I am responsible for every aspect of the game, but on this page I want to showcase what I did in regards to 4 areas: Gameplay Programming, Shader Programming, Game AI, and the use of the Gameplay Ability System. In developement, I leverage a mix of C++ and Blueprints, trying to implement as much of the heavy lifting in C++ as possible. But more on that below.

Manager Registry

ABSTRACTICA, despite its minimalistic art style, has grown into decently big project. I realized at some point that I was gonna have multiple distinct manager classes for spawn points, enemy locations, material management and other things. So, instead of turning all of the managers to their own singleton (which they sometimes shouldn't even be) I decided to go for a more controlled approach: The Manager Registry, a.k.a "Service Locator" or "Singleton Directory". 

UCLASS()
class ABSTRACTICA_API UMyManagerRegistry : public UGameInstanceSubsystem
{
	GENERATED_BODY()

	// Manager Registry Code ...
};

As a child of UGameInstanceSubsystem, the Manager Registry acts like it's own singleton, so I did not avoid using singletons entirely. But, I found this solution to be much cleaner than having every manager be it's own subsystem.

A new manager is usually registered on PostInitializeComponents(), which fires before on BeginPlay(). The registering method broadcasts the registration of a new manager, as well as clearing any pending requests for that particular manager that were made before the manager was registered.

void UMyManagerRegistry::RegisterManager(AActor* ManagerActor, TSubclassOf<AActor> ManagerClass)
{
	if (!ManagerActor)
	{
		return;
	}

	ManagerRegistry.Add(ManagerClass, ManagerActor);

	ManagerRegisteredEvent.Broadcast(ManagerActor);

	// Fire and clear any pending callbacks waiting for this manager
	if (TArray<TFunction<void(AActor*)>>* Callbacks = PendingCallbacks.Find(ManagerClass))
	{
		for (auto& Callback : *Callbacks)
		{
			Callback(ManagerActor);
		}
		PendingCallbacks.Remove(ManagerClass);
	}
}

void UMyManagerRegistry::UnregisterManager(AActor* ManagerActor, TSubclassOf<AActor> ManagerClass)
{
	if (!ManagerActor)
	{
		return;
	}

	ManagerRegistry.Remove(ManagerClass);
}

To get access to one of the managers, we need to request the manager class from the Manager Registry, which then checks if the manager is already registered or if we should wait for it to be registered. Either way, we wait until we get a valid response, which it then returned.

// Fires immediately if already registered, otherwise queues
template <typename T>
void RequestManager(TFunction<void(T*)> Callback)
{
	UClass* ManagerClass = T::StaticClass();

	if (const TWeakObjectPtr<AActor>* Found = ManagerRegistry.Find(ManagerClass))
	{
		if (Found->IsValid())
		{
			Callback(Cast<T>(Found->Get()));
			return;
		}
	}

	// Queue the callback, wrapped to cast to the correct type
	PendingCallbacks.FindOrAdd(ManagerClass).Add([Callback](AActor* Actor)
	{
		Callback(Cast<T>(Actor));
	});
}

This functionality helps keep a clean approach to the registration of managers, and is also very useful when triggering logic at very specific points. A good example of this in action is from the Level Loading Manager. In the example, I wait for the registered manager to be the level enemy spawn manager, to then do a different bit of logic right after the registration.

Enemy Spawn Manager

To pick up from the last example, I want to highlight what the Enemy Spawn Manager actually does. After the registration is done, which is the point at which we assume the level is loaded, a spawn offset timer start, to allow for the player to settle in for a couple of seconds before the enemies spawn and all hell breaks loose.

Each Level in ABSTRACTICA has it's own library of spawn points. These spawn points are stored in a level specific Enemy Spawn Manager. As described earlier, this manager is registered in PostInitializeComponents(). Through the use of the manager registry, we then request the spawn manager like this:

Because ABSTRACTICA is a roguelike, you will come across the same level multiple times. To vary the experience, I defined a random amount range of enemies per type that could potentially spawn. Each Enemy Spawn Point in a level also has an associated enemy type EMyShapeEnemyType, which we use to randomly select the wanted spawn points that we are going to activate. This makes sure, that I have full control over the potential spawn positions, while still keeping the procedural element of spawning alive.

TArray<AMyEnemySpawnPoint*> AMyLevelEnemySpawnManager::GetRandomAmountOfSpawnPointsByShapeEnemyType(
	const EMyShapeEnemyType shapeEnemyType,
	const int minRandomRange,
	const int maxRandomRange
)
{
	TArray<AMyEnemySpawnPoint*> spawnPoints;

	TArray<AMyEnemySpawnPoint*> filteredSpawnPoints;

	for (AMyEnemySpawnPoint* spawnPoint : enemySpawnPoints)
	{
		if (spawnPoint->GetShapeEnemyType() != shapeEnemyType)
		{
			continue;
		}

		filteredSpawnPoints.Add(spawnPoint);
	}

	const int32 clampedMax = FMath::Min(maxRandomRange, filteredSpawnPoints.Num());
	const int32 clampedMin = FMath::Min(minRandomRange, clampedMax);
	const int32 randomAmount = FMath::RandRange(clampedMin, clampedMax);

	TArray<AMyEnemySpawnPoint*> shuffled = filteredSpawnPoints;
	for (int32 i = shuffled.Num() - 1; i > 0; i--)
	{
		int32 j = FMath::RandRange(0, i);
		shuffled.Swap(i, j);
	}

	for (int32 i = 0; i < randomAmount; i++)
	{
		spawnPoints.Add(shuffled[i]);
	}

	return spawnPoints;
}

Shader Programming

One of the aspects of ABSTRACTICA that by far draws the most attention is the unique visual style. Inspired by the spongebob episode "SB-129" and a great lecture I heard about Signed Distance Fields, the game has a very simplistic, unlit-shaded world. To get a bit of juice into the game however, I wanted to build a shader that would allow me to directly influence the environment with differing objects, the most obvious of which is a bullet impact.

The "bullet impact tear drop"-effect works as follows: The position of the bullet hit is stored in a pre-loaded variable called IXY, with XY being the index in the limited array of tear drops we can register (16 in total). A bullet hit is registered through the General Material Manager to as follows:

void AMyGeneralMaterialManager::RegisterHit(const FVector& hitLocation)
{
	_hitDataArray[_currentHitIndex].isActive = true;
	_hitDataArray[_currentHitIndex].hitLocation = hitLocation;
	_hitDataArray[_currentHitIndex].intensity = 0.0f;

	_hitDataArray[_currentHitIndex].timer = 0.0f;

	_currentHitIndex++;

	_currentHitIndex %= _hitDataArray.Num();
}

_hitDataArray is an array of a custom struct FDisplayChangeSDFData, which keeps track of active tear drops in the shader. The Tick-method then handles the scaling of the radius that creates the tear drop effect. By using a UMaterialParameterCollection called displayModeParameterCollection, we can globally affect a fixed set of variables in a single class, and continously keep track of the active tear drops. We cycle through the hit data array and check, whether the element is currently active.

for (int i = 0; i < _hitDataArray.Num(); i++)
	{
		if (!_hitDataArray[i].isActive)
		{
			continue;
		}

If the element is active, we increase the timer on it. We then check if the timer exceeds the defined duration of the tear drop effect, and deactivate the element if necessary.

		_hitDataArray[i].timer += DeltaTime;

		_hitDataArray[i].isActive = _hitDataArray[i].timer  <= effectDuration;

If the element is still active, we then evaluate an effect curve by getting the ratio of the timer to the effect duration, which will give us a clean value between 0 and 1. The result is our new intensity of the tear drop, which will be set to the .A-parameter value of the impactParameter from the Material Parameter Collection.

		if (_hitDataArray[i].isActive)
		{
			const float eval = FMath::Clamp(_hitDataArray[i].timer / effectDuration, 0.0f, 1.0f);

			_hitDataArray[i].intensity = effectCurve->GetFloatValue(eval);

			const FName impactParameterName = FName(*FString::Printf(TEXT("I%d"), i));

			FLinearColor impactParameter;
			impactParameter.R = _hitDataArray[i].hitLocation.X;
			impactParameter.G = _hitDataArray[i].hitLocation.Y;
			impactParameter.B = _hitDataArray[i].hitLocation.Z;
			impactParameter.A = _hitDataArray[i].intensity;

			UKismetMaterialLibrary::SetVectorParameterValue(
				GetWorld(),
				displayModeParameterCollection,
				impactParameterName,
				impactParameter);
		}
	}

Using a custom material expression in Unreal Engines Material Editor, we can then feed all of the parameters, and a custom of the Material Parameter Collection into the following function:

float result = 0.0f;

float4 slots[16] = 
{ 
I0, I1, I2, I3, I4, I5, I6, I7,
I8, I9, I10, I11, I12, I13, I14, I15
};

for (int i = 0; i < 16; i++)
{
    float3 impactPos   = slots[i].xyz;
    float  intensity   = slots[i].w;      // driven by Blueprint timeline, 0→1→0

    float dist = length(WorldPos - impactPos);
    float edge = saturate(1.0 - (dist / ImpactRadius));
    edge = pow(edge, 2.0);
    result += edge * intensity;
}

return saturate(result);

The result is a mask on the material, which we use to display the alternative color of the material. Each material used in ABSTRACTICA has a, for lack of better naming, Dark Color and a Light Color. These colors are complimentary colors, which creates a nice harmonic contrast when a tear drop is triggered. The mask we created gets passed on to be multiplied with both the light and the dark color, and the result is the color of that fragment on the material.

A nice side effect of the use of Material Parameter collections is the fact that a tear drop "bleeds" between different surfaces that use different materials.

Gameplay Ability System

This is where the GAS stuff goes.

// Grant the ability to the ASC at begin play
void AAbstracticaCharacter::BeginPlay()
{
    Super::BeginPlay();

    if (HasAuthority() && AbilitySystemComponent)
    {
        for (TSubclassOf<UGameplayAbility>& Ability : DefaultAbilities)
        {
            AbilitySystemComponent->GiveAbility(
                FGameplayAbilitySpec(Ability, 1, INDEX_NONE, this));
        }
    }
}

// Activate by gameplay tag (called from input binding)
void AAbstracticaCharacter::ActivatePrimaryAbility()
{
    FGameplayTagContainer TagContainer;
    TagContainer.AddTag(FGameplayTag::RequestGameplayTag(FName("Ability.Primary")));
    AbilitySystemComponent->TryActivateAbilitiesByTag(TagContainer);
}
